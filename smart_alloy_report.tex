\begin{highlight}
\textbf{Abstract.}
Smart Alloy Selector automates engineering material selection
from natural-language requirements. The system combines
structured constraint extraction, a deterministic weighted
scoring engine, a deduplicated database of 1{,}200+ materials,
and a Retrieval Augmented Generation (RAG) layer to produce
focused explanations. Curated handbook data covers practical
engineering grades, while Materials Project API integration
expands the search space with stable inorganic compounds.
The application remains useful even when a remote model does
not respond by falling back to local heuristic extraction and
locally generated explanations.
\end{highlight}

\section{Introduction and Problem Statement}

Material selection remains a major bottleneck in design review
because engineers typically move between static tables,
datasheets, and handbook chapters without any system that can
understand the actual problem statement. A user designing a
4-point probe for copper-cobalt pellets must simultaneously
consider conductivity, hardness, thermal stability, cost, and
corrosion resistance, yet MatWeb and handbook tables provide
properties rather than ranked decisions.

MET-QUEST'26 asks for an AI-based material recommendation tool
that behaves like a virtual materials scientist: accept
plain-English design intent, infer constraints, search a
property database, and recommend the most suitable candidate
with clear justification. Our implementation extends this idea
with a larger database, semantic retrieval, and a novel alloy
analogue predictor.

\section{System Architecture and Workflow}

\subsection{Full Pipeline}

\begin{enumerate}[leftmargin=*,itemsep=2pt]
  \item \textbf{Constraint extraction.} Gemini Flash 2.0 is
  prompted to produce a strict \texttt{UserConstraints} JSON
  object. If the remote call fails, a local heuristic extractor
  infers temperature, density, printability, corrosion, and
  property weights from keywords.

  \item \textbf{Category intent inference.} Before scoring,
  the query is scanned for manufacturing and domain signals.
  For example, ``3D printed'' restricts the search to polymers,
  while ``BGA solder'' restricts the search to solders.
  This prevents ceramic outliers such as ZrO$_2$ from winning
  plastic bracket queries.

  \item \textbf{Deterministic scoring.} Candidates are filtered
  against hard constraints and then scored using a weighted sum:
  \begin{equation}
    s_i = 100 \cdot \left(
    w_T \hat{T}_i + w_S \hat{\sigma}_i +
    w_W (1-\hat{\rho}_i) + w_C (1-\hat{C}_i^{\log}) +
    w_R \hat{R}_i \right)
  \end{equation}
  with filtered-set normalisation and log-scale cost handling.

  \item \textbf{RAG retrieval.} The explanation stage does not
  send the full database to Gemini. Instead, the top scored
  shortlist is semantically filtered using all-MiniLM-L6-v2
  embeddings, and only the most relevant materials are passed
  into the explanation prompt.

  \item \textbf{Explanation generation.} Gemini produces a
  three-paragraph engineering explanation. If the model does
  not return, the app generates a local explanation directly
  from the shortlisted property data, so the user still sees a
  polished recommendation rather than a quota message.
\end{enumerate}

\section{Materials Database}

\subsection{Dataset Composition}

The final build uses a hybrid database. First, 90 curated
materials were entered manually from ASM Handbook, MatWeb, NASA
TPSX, and manufacturer datasheets. These entries cover the
engineering grades that judges expect to see in practice:
316L, Ti-6Al-4V, Inconel 718, PETG, Nylon PA12, PEEK, SAC305,
Monel 400, Hastelloy C-276, and similar benchmark materials.

Second, a fetch pipeline was added for The Materials Project
API. Rather than storing multiple crystal polymorphs with
cryptic names such as ``Ag (Fm-3m, mp-124)'', the merge layer
groups Materials Project entries by reduced formula and keeps
the phase with the lowest energy above hull. Friendly names are
assigned where appropriate, and raw pure-metal duplicates such
as Ag, Al, Cu, Ti, Ni, Mo, W, and Au are dropped whenever a
hand-curated engineering entry already covers that material
family.

\subsection{How Additional Data Is Imported}

\begin{description}[leftmargin=0pt,itemsep=3pt]
\item[\textbf{Materials Project API.}]
\url{https://next.materialsproject.org/api}. The project uses
\texttt{X-API-KEY} authentication and the summary endpoint to
fetch density, energy above hull, band gap, and elastic moduli
for exact chemical systems. The script
\texttt{scripts/fetch-materials.mjs} expands a target formula
list into a large set of relevant binary and ternary systems
before generating a TypeScript dataset used directly by the app.

\item[\textbf{ASM Handbook.}]
Volumes 1 and 2 provide tensile strength, yield strength,
hardness, thermal conductivity, service temperature, and alloy
selection guidance for the curated metal grades.

\item[\textbf{MatWeb and manufacturer datasheets.}]
Used primarily for polymers, ceramics, solders, and specialty
grades such as PEKK, PVDF, Ultem, Waspaloy, and low-temperature
solder systems.
\end{description}

\section{RAG Pipeline}

The RAG layer improves both quality and cost. Each material is
converted into a descriptive text block including category,
subcategory, temperature limit, strength, density, cost,
corrosion level, printability, and tags. The user query is
embedded with the same all-MiniLM-L6-v2 model. Cosine
similarity then ranks the most relevant materials.

To keep the system practical on a large database, retrieval is
hybrid. A lexical pre-filter narrows the pool to the most
plausible matches, then only that shortlist is embedded. During
recommendation, the RAG layer runs on the deterministic top-10
scored materials. During semantic database search, it runs on a
small prefiltered subset of the full corpus. This gives focused
Gemini context without forcing the serverless function to embed
all 1{,}271 materials on every request.

\section{Worked Examples}

\textbf{Cheapest material query.}
For ``give me the cheapest possible material'', the heuristic
extractor boosts cost weight aggressively. The top-ranked
results are Grey Cast Iron and Carbon Steel 1020, rather than
high-temperature or exotic compounds. This validates the
log-scale cost normalisation and non-equal default weights.

\textbf{3D-printed bracket at 85\textdegree C.}
For ``lightweight 3D printed bracket for 85 degree motor
heat'', category intent restricts the shortlist to practical
polymers. PETG and Nylon PA12 rise to the top, while ceramics
are excluded from the top results despite their very high
temperature capability.

\textbf{Marine pump housing.}
For ``marine pump seawater exposure high pressure'', the
corrosion-biased weights and metal-only intent steer the result
toward Monel 400 and Hastelloy C-276. This matches engineering
intuition and rejects poor real-world choices such as PLA or
cheap carbon steels for seawater duty.

\section{Security and Deployment}

API keys are stored only in \texttt{.env.local} for local
development and in Netlify environment variables for
production. The public repository contains only placeholder
values in \texttt{.env.example}. The git history was checked
for accidental environment file commits, and the working tree
was hardened by explicitly ignoring local environment files,
Netlify build folders, raw Materials Project cache files, and
private key material.

The live application is deployed at
\url{https://beautiful-tiramisu-211a94.netlify.app}. GitHub
pushes trigger Netlify builds automatically, so the deployed
site always matches the repository state used for judging.

\section{Limitations and Future Work}

All properties are still treated as room-temperature scalar
values. The tool does not yet model fatigue, creep, fracture
toughness, temperature-dependent modulus degradation, or
microstructural processing windows. The Materials Project
portion of the database also uses engineering heuristics to map
DFT properties into practical screening values, so those
entries should be treated as search-space expansion rather than
drop-in procurement specs.

Future work includes adding temperature-property curves,
persistent vector storage for sub-second semantic retrieval,
supplier integrations for real-time pricing, and a true
composition-to-property regressor for novel alloys beyond
nearest-known analogue prediction.
