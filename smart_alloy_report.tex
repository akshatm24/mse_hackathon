\begin{highlight}
\textbf{Abstract.}
Smart Alloy Selector is a natural-language materials recommendation system that combines structured query extraction, deterministic ranking, shortlist-grounded explanation generation, and a split-database novel-composition predictor. The final expansion in this repository now serves 7,118 unique materials through two runtime pools: 1,382 curated engineering grades for the main selector and 5,736 Materials Project compounds for the novel-alloy workflow. The ranking layer was updated for duplicate-ID cleanup, engineering-only filtering, property-specific match reasons, deterministic tie-breaking, turbine-blade intent handling, null-safe UI rendering, and a visible predictor workflow that maps computed compounds to engineering analogues. The resulting build passes TypeScript, Jest, API smoke, and production-build checks and is deployed on Netlify.
\end{highlight}

\section{Problem Statement}

The MET-QUEST'26 challenge asks for a virtual materials scientist: accept a plain-English engineering query, infer practical constraints, search a large property database, and return a ranked recommendation with explanation. In practice, this requires more than a semantic search layer. The system must:

\begin{itemize}[leftmargin=*]
  \item remain stable when important properties are missing,
  \item interpret negation correctly,
  \item reject obviously wrong categories for task-specific queries,
  \item scale to thousands of rows without breaking the UI,
  \item and stay secure while using external APIs.
\end{itemize}

\section{System Architecture}

Each recommendation request follows the same five-stage pipeline:

\begin{enumerate}[leftmargin=*]
  \item \textbf{Constraint extraction.} Gemini plus a local heuristic fallback convert the query into a \texttt{UserConstraints} object.
  \item \textbf{Intent inference.} Query phrases such as \texttt{BGA solder}, \texttt{3D print}, \texttt{cryogenic}, \texttt{marine}, and \texttt{biomedical implant} activate category and property gates.
  \item \textbf{Deterministic ranking.} Candidates are hard-filtered and scored with null-safe logic.
  \item \textbf{Shortlist-only RAG.} Explanation context is chosen only from ranked candidates, preventing explanation drift.
  \item \textbf{Response generation.} The API returns ranked cards, warnings, inferred constraints, and explanation text.
\end{enumerate}

This architecture keeps the explanation aligned with the actual returned shortlist and prevents semantic retrieval from introducing materials that the scorer never selected.

\section{Database Expansion Pipeline}

\subsection{Phase 0 Audit}

The checked-out baseline in this repository contained 655 existing materials. That baseline was preserved and treated as the highest-priority curated layer for all subsequent merges.

\subsection{Materials Project Expansion}

The Materials Project stage was rebuilt around paginated fetching instead of a hardcoded formula list. The fetch script queries stable metals, stable non-metals, and low-hull metastable phases, then joins elasticity data in secondary batches. Engineering relevance filters exclude radioactive or clearly impractical systems and prefer known engineering families such as:

\begin{itemize}[leftmargin=*]
  \item engineering metals,
  \item oxides, carbides, nitrides, borides, and silicides,
  \item semiconductor compounds,
  \item fluorides used in optical applications,
  \item intermetallics with known engineering relevance.
\end{itemize}

The processing stage converts MP rows into the runtime material format, estimates only defensible screening properties, enriches existing rows without deleting them, and keeps deduplicated formula-pretty representatives.

\subsection{Web Scraping and Cited Sources}

The data pipeline also includes scrapers and hardcoded cited datasets for:

\begin{itemize}[leftmargin=*]
  \item EngineeringToolbox tables,
  \item Wikipedia property tables,
  \item manufacturer alloy pages,
  \item NIST enrichment hooks,
  \item best-effort AZoM and MatWeb stages,
  \item hardcoded cited entries for key engineering grades.
\end{itemize}

The hardcoded layer is used where reliable engineering data is available from datasheets or handbook-style references but not easy to scrape consistently.

\subsection{Final Assembly}

The final assembly order is:

\begin{enumerate}[leftmargin=*]
  \item preserved curated baseline,
  \item hardcoded cited entries,
  \item scraped entries,
  \item Materials Project entries.
\end{enumerate}

After assembly, the runtime dataset is deduplicated by material ID and split into \texttt{src/data/materials.json} for engineering grades and \texttt{src/data/mp\_materials.json} for Materials Project compounds. Only null fields are enriched from lower-priority sources. Existing curated values are never overwritten by weaker sources.

\section{Scoring and Query-Handling Fixes}

The final ranking update focused on making the recommendation layer robust for real engineering prompts.

\subsection{Null-Safe Scoring}

Missing properties no longer crash or silently bias results:

\begin{itemize}[leftmargin=*]
  \item if a primary property for the current query is missing, the material is removed from the strict candidate pool for that constraint,
  \item if only a secondary property is missing, that subscore becomes neutral instead of punitive,
  \item if the strict pool collapses to zero, a relaxed fallback keeps category intent and domain gates while dropping only the most restrictive numeric limits.
\end{itemize}

\subsection{Negation and Task-Specific Intent}

The ranker now explicitly handles:

\begin{itemize}[leftmargin=*]
  \item \texttt{cheap}, \texttt{budget}, \texttt{cost irrelevant},
  \item \texttt{lightweight}, \texttt{weight does not matter},
  \item \texttt{no ceramics}, \texttt{not polymer},
  \item \texttt{not brittle}, \texttt{non-magnetic},
  \item \texttt{cryogenic}, \texttt{liquid nitrogen},
  \item \texttt{biomedical implant},
  \item \texttt{BGA solder joint 220C reflow, lead-free},
  \item \texttt{heat sink, conduct heat away fast}.
\end{itemize}

One important correction was splitting \emph{service temperature} from \emph{thermal conductivity}. Generic heat-sink prompts now rank by heat-flow performance rather than by refractory temperature capability. Reflow solder prompts likewise use melting-window suitability instead of service-temperature ranking.

Two additional reliability fixes were critical in the final pass:

\begin{itemize}[leftmargin=*]
  \item the response is deduplicated immediately before return, so repeated IDs never appear in the UI or API payload,
  \item turbine / hot-section queries now explicitly prefer nickel superalloys and related blade-capable materials over brittle refractory ceramics.
\end{itemize}

\subsection{Novel Composition Prediction}

The predictor now uses the dedicated Materials Project compound pool instead of the main recommendation set. A formula such as \texttt{NiCo20Cr20MoTi} is parsed into element fractions, matched against the closest MP compound by cosine similarity, and then mapped to the nearest curated engineering analogue for practical recommendation. The response is explicitly labeled as screening guidance rather than a certification-grade prediction.

\section{UI and Runtime Changes}

The interface was updated for the larger dataset:

\begin{itemize}[leftmargin=*]
  \item the hero and stats bar now show the live material count,
  \item the database browser defaults to engineering grades and exposes an explicit Materials Project toggle,
  \item the runtime data is served from split JSON pools rather than one monolithic client-bundled import,
  \item null values render as em-dash placeholders instead of fake zeros,
  \item source and data-quality fields are surfaced in the UI layer,
  \item recommendation cards now include per-axis score breakdowns and CSV export,
  \item the main page includes an Ashby plot tab for density-versus-strength exploration,
  \item detail sheets expose clickable source links when a datasheet URL exists.
\end{itemize}

\section{Security and Operational Notes}

Security requirements were enforced through the workflow:

\begin{itemize}[leftmargin=*]
  \item local secrets remain in \texttt{.env.local},
  \item source code was checked for hardcoded keys,
  \item generated raw scrape files and local environment files are ignored by git,
  \item the README documents the pipeline without exposing secrets.
\end{itemize}

\section{Conclusion}

The final Smart Alloy Selector build is no longer a small curated demo. It is a multi-source engineering screening system with a 7,118-row runtime database, deterministic ranking, shortlist-grounded explanations, and a split data architecture that keeps curated engineering grades separate from Materials Project compounds. The resulting app is substantially broader in coverage while also being safer: it handles missing data, removes duplicate IDs, avoids common query-parsing traps, exposes richer UI diagnostics, and remains auditable from raw source ingestion to final recommendation output.
